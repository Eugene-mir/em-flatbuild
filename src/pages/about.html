em-Flatbuild

A modern flat file generator for rapid website development. Inspired by Panini, but standalone - no Gulp required.

Handlebars templating SCSS compilation Markdown pages Live reload dev server

Quick Start


npx em-flatbuild dev   # Start dev server with live reload
npx em-flatbuild build --minify  # Build for production

Place your source files in the src/ directory, then run dev and open http://localhost:3000.

Project Structure


my-project/
├── src/
│   ├── pages/          # .html / .hbs / .md pages with front matter
│   ├── layouts/        # Layout templates (default.html required)
│   ├── includes/       # Reusable HTML fragments
│   ├── helpers/        # Custom Handlebars helpers (.js)
│   ├── data/           # Global data - JSON / YAML / JS
│   ├── assets/         # Static files (JS, images, fonts)
│   └── scss/           # SCSS stylesheets
├── dist/               # Build output (generated)
└── em-flatbuild.config.js # Configuration (optional)

Pages

Pages live in src/pages/ and support .html, .hbs, .handlebars, and .md extensions. Every page can include YAML front matter at the top: 


---
title: About Us
layout: default
customVar: hello
---

<h1>{{title}}</h1>
<p>{{customVar}}</p>

Front matter variables are available as Handlebars context inside the page and its layout.

The page Object

Every page template receives a page object with metadata:

  • page.name - filename without extension
  • page.path - relative path
  • page.title - from front matter

Layouts

Layouts define the outer HTML shell that wraps page content. Place them in src/layouts/. Use the special partial to inject the page content.

A layout named default.html is required - it's the fallback for all pages. Create additional layouts (e.g. blog-post.html, landing.html) and reference them in front matter.

Includes

Reusable HTML fragments in src/includes/. Any .html or .hbs file is auto-registered as a Handlebars partial. 


src/includes/
├── header.html
├── footer.html
└── cards/
    └── product.html

Nested files use forward-slash naming. Includes receive the same data context as the calling template.

Data

External data files in src/data/ become global template variables available in every page, layout, and include.

Format Example file Template access
JSON site.json {{site.name}}
YAML team.yml {{#each team}}...{{/each}}
JavaScript build.js {{build.date}}

Pagination

Any list page can be split into multiple static HTML pages by adding two front matter keys: paginate and perPage. 


---
title: Blog
layout: blogs
paginate: blogs
perPage: 10
---

Output Structure

Given src/pages/blogs.html with 25 items and perPage: 10, the build produces: 


dist/
├── blogs.html        ← page 1
├── blogs/
│   ├── 2.html        ← page 2
│   └── 3.html        ← page 3

The pagination Object

Every paginated page receives a pagination object:

  • pagination.current - Current page number (1-based)
  • pagination.total - Total number of items
  • pagination.totalPages - Total number of pages
  • pagination.perPage - Items per page
  • pagination.prev - URL of previous page or null
  • pagination.next - URL of next page or null

Markdown

Markdown Pages

.md files in src/pages/ are first-class citizens. Handlebars expressions are processed first, then the result is converted to HTML via marked and wrapped in the layout.

The pipeline: Front matter parsed → Handlebars compiled → Markdown → HTML → Layout wrapped → Output.

Markdown Helper

Use the markdown helper to render markdown inside HTML pages in both block and inline modes.

Helpers

Custom Helpers

Place .js files in src/helpers/. Each file should export a single function (ESM export default). The filename becomes the helper name. 


// src/helpers/uppercase.js
export default function (str) {
  if (typeof str !== 'string') return '';
  return str.toUpperCase();
};

Built-in Helpers

Helper Type Description
eq, neq, gt, lt Block Conditional comparisons
and, or Block Logical operators
repeat Block Repeat N times
markdown Both Markdown to HTML
icon Inline Inline SVG icon
join Inline Array to string
json Inline Pretty JSON
year Inline Current year

Icons

SVG icons placed in src/assets/icons/ can be inlined directly into HTML using the icon helper. The SVG source is injected as raw HTML with semantic class names - no sprite sheets, no extra HTTP requests.

Icons are also copied to dist/assets/icons/ as static files, so they remain available via URL if needed.

SCSS

SCSS files in src/scss/ are compiled to CSS automatically. Output goes to dist/css/.

Files starting with _ (partials) are not compiled as standalone - they're meant to be imported. Use --minify flag on build to get compressed output. 


src/scss/
├── main.scss          →  dist/css/main.css
├── _variables.scss    →  @import only
└── _reset.scss        →  @import only

Assets

Static files in src/assets/ are copied as-is to dist/assets/, preserving directory structure. 


src/assets/
├── js/
│   └── main.js        →  dist/assets/js/main.js
├── images/
│   └── logo.svg       →  dist/assets/images/logo.svg
└── fonts/
    └── inter.woff2    →  dist/assets/fonts/inter.woff2

Linting

The project includes linter configs tuned for BEM methodology:

  • ESLint - JavaScript linting (flat config, ESM)
  • Stylelint - SCSS linting + BEM class naming
  • Prettier - Code formatting (JS, SCSS, HTML, JSON, MD)
  • EditorConfig - Consistent editor settings

Configuration

Create em-flatbuild.config.js in the project root. All options are optional: 


export default {
  // Paths (relative to project root)
  root: 'src/pages',
  layouts: 'src/layouts',
  includes: 'src/includes',
  helpers: 'src/helpers',
  data: 'src/data',
  assets: 'src/assets',
  scss: 'src/scss',
  output: 'dist',

  // Dev server
  port: 3000,
  open: false,       // Open browser on dev

  // Auto-assign layouts by folder
  pageLayouts: {
    // 'blog': 'blog',
    // 'docs': 'docs',
  },
};

CLI Reference


Usage: em-flatbuild <command> [options]
Commands:
  build [options]     Build the site for production
  dev [options]       Start dev server with live reload

Build options:
  --minify            Minify CSS output

Dev options:
  -p, --port <number> Dev server port (default: 3000)
  --open              Open browser on start

General:
  -V, --version       Show version number
  -h, --help          Show help

Examples


# Dev server on custom port
em-flatbuild dev -p 8080

# Dev server, open browser
em-flatbuild dev --open

# Production build with minified CSS
em-flatbuild build --minify

Programmatic API

em-Flatbuild can also be used as a Node.js library:


import { resolveConfig, build, serve } from 'em-flatbuild';
// Build
const buildConfig = await resolveConfig(process.cwd(), { minify: true });
const result = await build(buildConfig);
console.log(`Built ${result.pages} pages in ${result.elapsed}ms`);

// Dev server
const devConfig = await resolveConfig();
await serve(devConfig);

Key Features

  • Zero Configuration: Works out of the box with sensible defaults. Add config only when you need to customize.
  • Fast Build Times: Optimized for speed. Build hundreds of pages in seconds.
  • Developer-Friendly: Clean API, clear documentation, predictable behavior. Spend time building, not debugging tools.
  • Production-Ready: Minification, optimization, and best practices built in. Deploy with confidence.
  • BEM Methodology: Built-in linting support for BEM naming conventions.
  • Flexible Data: Support for JSON, YAML, and JavaScript data files.

Who It's For

em-Flatbuild is perfect for developers who need to build static websites quickly without the overhead of heavy frameworks. Whether you're creating marketing sites, landing pages, documentation, portfolios, or blogs - if it's content-driven and doesn't require a database, em-Flatbuild is an ideal choice.
